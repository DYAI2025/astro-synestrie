## Context
New_Bazi has a fail-closed Plumbline (PRIL) Stop hook (`~/.claude/agents/config/claude/hooks/plumbline-enforce.sh`) that blocks the session if changed files fall outside the active feature's allowed scope. This caused a mid-session block once. Run this BEFORE committing or finishing to catch it early.

## Your Task

Verify the PRIL scope gate will pass (or explain how to make it pass) for the current changes.

### Steps

1. **Find the REAL active-feature marker** — `docs/context/.active-feature` (NOT the root `.active-feature`, which is stale and ignored by the hook):
   ```bash
   cat docs/context/.active-feature 2>/dev/null || echo "ABSENT → hook no-ops, nothing to check"
   ```
   Absent marker = hook is a no-op (normal non-feature session). Done.

2. **Resolve the allowed scope** for that feature slug, in the hook's precedence order:
   - canvas heading literally named `Allowed change scope` in `docs/canvas/<feat>.canvas.md`, else
   - `docs/traceability.md` lines mentioning the feature + "scope", else
   - `docs/scope/<feat>.scope.json` (`allowed_change_scope` list).
   Note: a canvas that puts its scope under a `## N. Traceability` heading is NOT read — only a literal `Allowed change scope` heading or the scope JSON.

3. **Compute the changed-files surface** exactly as the hook does (merge-base(HEAD,main)..HEAD ∪ working-tree ∪ staged ∪ untracked-non-ignored) and run the gate:
   ```bash
   base="$(git merge-base HEAD main 2>/dev/null)"; [ -n "$base" ] || base=HEAD
   { git diff --name-only "$base"...HEAD; git diff --name-only; git diff --name-only --cached; git ls-files --others --exclude-standard; } | sort -u > /tmp/pril-changed.txt
   config/claude/bin/plumbline-scope-check --repo . --feature "<feat>" --changed-files /tmp/pril-changed.txt; echo "EXIT: $?"
   ```

4. **If it fails (exit 3 = violation, 2 = missing scope):** identify the out-of-scope files. Then choose the honest resolution and confirm with the user — do NOT silently broaden scope:
   - Files legitimately belong to the active feature → add concrete patterns (with path anchors) to `docs/scope/<feat>.scope.json`.
   - The active feature already shipped/merged → the marker is stale; clear `docs/context/.active-feature` (hook then no-ops). Verify the feature is actually merged first.
   - The work belongs to a different feature → repoint the marker (also check the context gate passes for the new slug).

5. **Confirm green** by re-running the gate, and optionally run the full hook:
   ```bash
   printf '{}' | CLAUDE_PROJECT_DIR="$PWD" bash ~/.claude/agents/config/claude/hooks/plumbline-enforce.sh; echo "EXIT: $?"
   ```
   Empty stdout = no block.

### Guardrails
- The REAL marker is `docs/context/.active-feature`. The root `.active-feature` is stale — ignore it.
- Never broaden scope with a wildcard that legitimizes the whole repo — the guard rejects patterns without a concrete path segment.
- Clearing the marker or repointing it is a governance action — confirm with the user; don't do it silently to dodge the gate.
- A failing gate must be fixed or escalated — never finish on a failing gate.
