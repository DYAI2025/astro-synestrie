## Narrative — the biggest completion gaps blocking user value

1. **The honesty backstop for Unknown-Time is the single largest *completion* gap masquerading as "done."** FR-015 *looks* finished (toggle + validation + provisional short-circuit all merged, which is why the ledger is stale), but the suppression of ascendant/houses/hour pillar is driven **100% by the engine's `provisional_fields`** (`fufireNormalizer.ts:230-233,440`). The locally-known `timeKnown` flag is read at line 229 and only echoed to output — it never forces suppression. Any engine version skew, a fallback-local path, or an endpoint that ignores `birth_time_known` makes the app render a **12:00-computed ascendant, 12 houses and an hour pillar as fact** — the exact anti-reification violation the product principle and the wording-honesty tests exist to prevent, and it is **unguarded by any test** (`BIRTH-TIME-01`). A feature is not "done" while its core honesty contract has no local guarantee.

2. **The ledger itself is a completion-planning hazard.** B-001, B-002, B-006/FR-007, B-004/FR-015 and especially FR-011 are marked open/missing but are DONE in code (`GAP-LEDGER-STALE-01`, `GAP-FR011-01`). Anyone sprinting from sprint-plan-v2 would **re-implement a finished Partner Journey** (`PartnerJourney.tsx`, `interAspects.ts`, `baziCompare.ts` already render four honest layers). The ledger must be re-baselined before any sprint is planned off it.

3. **The fake-zero class (B-002) was fixed for Fusion but not for WuXing.** An empty/garbage WuXing vector still renders five "Defizit" bars at 0% as `available` (`FE-WUXING-01`, `FUFIRE-WUXING-04`) — a fabricated "everything is deficient" reading. This is the same honesty bug the team already knows how to fix; it just wasn't applied to the parallel section.

4. **The real PDF report (FR-008) remains genuinely unbuilt** (`Overview.tsx:203` bare `window.print()`) — the ledger is correct here, and it is the largest *unstarted* user-facing feature among the in-scope findings.

5. **The persistence backend out-runs the UI.** FR-001's tables and 6 auth routes ship, but 4 of 6 routes (partner CRUD + profile delete) have **no client caller** (`API-GAP-06`), and the partner table is schema-asymmetric (`DB-INTEGRITY-06`, `GAP-DB-PARTNER-01`). Users can save a profile but never delete it; the entire partner-persistence layer is dead weight from the front end.
