## Executive summary
New_Bazi's engineering core is healthy — 621 vitest pass, tsc clean, and the heaviest features the ledger lists as "missing" (FR-011 Synastry pair journey, FR-007 explanation layers, B-001 coaching wording, B-002 coherence missing-state) are in fact DONE in code, so the 2026-06-11 sprint-plan-v2 ledger is stale and must be re-baselined before any sprint is planned off it. Across the 52 verified findings the spread is roughly 8 high-severity, ~16 medium, and ~28 low/uncertain, with the high band concentrated in two themes: anti-reification honesty leaks and unmetered/un-isolated server surface. The three things to do first: (1) give Unknown-Time a *local* honesty backstop — today suppression of the 12:00 ascendant/houses/hour pillar depends 100% on the engine's `provisional_fields` and `BIRTH-TIME-01` shows a 12:00-derived chart can render as fact, the exact violation the product principle forbids; (2) add rate-limiting/auth to the unauthenticated compute routes (`SEC-RATELIMIT-01`) which today allow unbounded billable Gemini/FuFirE spend; (3) switch the profile orchestrator from `Promise.all` to `allSettled` (`FUFIRE-ORCH-01`) so one transient upstream blip stops 502-ing the entire profile. The honesty/anti-reification principle is mostly intact at the shipped-engine layer but has real, unguarded cracks: the present-but-empty WuXing fake-0% bars (`FE-WUXING-01`/`FUFIRE-WUXING-04`), the fabricated local-fallback retrograde/longitude math (`ASTRO-MATH-01/02/07`), a hardcoded "1.5x Wichtung" metric in BaZiDetail (`FE-BAZI-METRIC-01`), "Haus 0" leaking as a real house (`FE-WEST-HOUSE-01`), and a `signalLevel` emitted over a null coherence (`FUFIRE-HONESTY-02`) — all clustered in detail components and the local-fallback path that the wording-honesty scanners do not cover, so closing those scanner gaps is the highest-leverage way to keep the principle enforced rather than aspirational.

## Top risks
1. Anti-reification honesty is enforced by tests only on the landing/shell — detail components and the entire local-fallback astrology path are unscanned, so fabricated metrics (1.5x Wichtung, retrograde/longitude formulas, Haus 0, 0% WuXing 'Defizit' bars, signalLevel over null coherence) ship as fact; the single worst case is BIRTH-TIME-01 where a 12:00-derived ascendant/houses/hour pillar can render as fact whenever the engine's provisional_fields are absent, because the local timeKnown flag never drives suppression.

2. All expensive compute routes (/api/gemini/reading, /api/azodiac/{synastry,daily,profile}) are anonymous and unthrottled, exposing unbounded billable Gemini-token and FuFirE-quota spend (denial-of-wallet) with no rate-limit package in dependencies.

3. Tenant isolation rests entirely on every handler manually appending .eq('user_id') against a service-role client that bypasses RLS by design — one future query that forgets the filter leaks or mutates cross-tenant rows with zero DB backstop, and no test would catch it because tests mock the Supabase client.

4. Resilience is brittle at the FuFirE boundary: Promise.all makes profile build all-or-nothing (one 429/503 fails the whole profile) and the client has no retry/backoff — a single transient blip degrades the core read instead of rendering honest per-section missing-state.

5. The sprint-plan-v2 ledger over-states open work (FR-011 Synastry, FR-007/B-006 layers, B-001, B-002, B-004/FR-015 are DONE in code) and under-states the BIRTH-TIME-01 honesty regression — planning from it risks both re-doing finished features and missing the live anti-reification crack.

6. Deploy/config traps fail silently: a boot failure exits 0 (defeating ON_FAILURE restart), VITE_SUPABASE_* build args are unwired with no guard (login silently dies if creds absent), and malformed JSON bodies leak full stack traces with absolute repo paths to the client.

Three independent authoritative sources contradict it: (1) the installed SDK @google/genai@2.6.0 (satisfies the package.json constraint ^2.4.0) explicitly enumerates 'gemini-3.5-flash' in its Model_2 TypeScript union type in node_modules/@google/genai/dist/node/node.d.ts, alongside other Gemini 3.x ids (gemini-3-flash-preview, gemini-3.1-*); the SDK the code actually uses treats this id as valid. (2) Official Google docs (ai.google.dev/gemini-api/docs/interactions/whats-new-gemini-3.5) state Gemini 3.5 Flash is generally available, stable, and production-ready, doc-updated 2026-06-04. (3) Web search confirms GA release on 2026-05-19. The environment date is 2026-06-15, well after release. The finding author reasoned from a pre-2026 knowledge cutoff (when valid flash ids were only 1.5/2.0/2.5) and did not verify against current model availability or the installed SDK. The code at app.ts:509 does use this model id (that part of the report is accurate), and the .env file does contain a real GEMINI_API_KEY making the configured path reachable — but that path would SUCCEED, not fail. The secondary 'zero test coverage' sub-claim is literally true (rg -li gemini over test files = 0 hits; no test exercises the configured-success or error branch), but the finding frames this as evidence that a defect 'was never caught' — there is no defect to catch, so the missing tests masked nothing. Untested optional Gemini endpoint is at most a minor hygiene gap, not the medium/medium bug described. The finding as written (invalid model id, guaranteed failure, masked outage) is refuted.
- **API-INPUT-07** /api/places/autocomplete forwards empty/missing input upstream; no min-length guard (asymmetric with client) — The finding's central, explicitly-stated claim is false. It asserts that `{}` or `{input:''}` "issues a real upstream Photon query for an empty string" and that "Empty string [is] forwarded to the Photon provider on every empty/missing-input request." It is not. The handler at src/server/app.ts:323 does `getAutocompletePredictions(req.body?.input || "")`, so both `{}` and `{input:''}` yield `""`. But `getAutocompletePredictions` (src/utils/mapsService.ts:137-140) opens with its own guard: `if (!input || input.trim().length === 0) return [];` — this short-circuits and returns `[]` BEFORE the Photon `fetch` at line 148. The author reasoned only from app.ts and the client guard (bazodiacClient.ts:258) and missed the server-side util guard. This is independently confirmed by an existing PASSING test, src/utils/mapsService.test.ts:75-80, which asserts `getAutocompletePredictions("")` resolves to `[]`. So the empty/missing-input contract IS enforced server-side, with no upstream call — directly contradicting the finding's "actual" and "evidence." A residual, much narrower truth exists: the server guard is `length === 0` while the client guard is `< 2`, so a single non-whitespace char (e.g. "a") passes the server but is blocked client-side, meaning a direct caller could trigger a 1-char Photon query. However that 1-char asymmetry is NOT what the finding describes (its title/expected/actual/evidence all center on empty/missing input, which is provably handled), the provider is keyless/free with no stated quota cost, and Photon handles short queries fine. The finding as written is contradicted by the code and an existing test.
